# Contributing to the Constitutional Agent SDK

Thank you for your interest in contributing. This project is in active
pre-release development. Contributions are welcome across code,
documentation, theoretical review, and empirical validation.

---

## Current Status

The SDK is in pre-release. The underlying architecture has been validated
in three deployed systems (AURA-ECHO, DNI, WFF) but the SDK abstraction
is in active development. The most valuable contributions right now are:

* **Theoretical review** — feedback on the physics grounding and
  architectural decisions
* **Documentation improvements** — clarity, corrections, additions
* **Issue reports** — if you find errors in the empirical claims or
  theoretical derivations, please open an issue immediately
* **Integration examples** — if you have tested the concepts in your
  own system, we want to know

---

## How to Contribute

### Reporting Issues

Open an issue on GitHub. Please include:

* Which document or component the issue relates to
* The specific claim or code that appears incorrect
* Your proposed correction with supporting evidence or reasoning

### Documentation Contributions

1. Fork the repository
2. Create a branch: `git checkout -b docs/your-description`
3. Make your changes
4. Submit a pull request with a clear description of what you changed
   and why

### Code Contributions

The TypeScript and Python SDK components are in development. When they
are ready for external contribution, this section will be updated with
specific guidance.

## Theoretical Contributions

The Constitutional Agent SDK is grounded in three theoretical frameworks.
If you have expertise in:

* **Aitiopoietic Cognition** (contact Dr. Tomas Veloz)
* **Causal Saturation Theory / Grammar of Stability** (contact Matthew Prager)
* **Constitutional Physics** (contact Dr. Carlos Arleo)

and you have substantive theoretical feedback, please reach out directly
before opening a public issue.

---

## Standards

All contributions must:

* Be honest about what is validated vs what is in development
* Not introduce claims unsupported by the empirical record
* Maintain the clarity and accessibility of the documentation for
  both technical and non-technical readers
* Respect the dual MIT/Apache 2.0 licence
